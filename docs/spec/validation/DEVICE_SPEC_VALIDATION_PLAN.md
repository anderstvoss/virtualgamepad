# Device Spec Validation Plan

This document defines how to discover, validate, and maintain the actual device specifications that virtual gamepad profiles must follow.

The goal is to minimize reverse-engineering effort by exhausting public sources first, turning any remaining captures into reusable fixtures, and requiring each profile to state exactly what evidence supports its claimed behavior.

## Evidence ladder

Use the least invasive evidence source that can prove the required behavior.

1. official specifications, vendor documentation, published descriptor data, or standards documents
2. operating-system drivers, especially Linux kernel drivers and public HID input mappings
3. open-source controller drivers, emulators, protocol libraries, and community test tools
4. host-visible descriptors and reports captured from a real device
5. USB, Bluetooth, or bus-level traces from a real device
6. black-box behavioral comparison against target software

Reverse engineering should start only after public sources and open implementations fail to answer a concrete validation question.

## Profile evidence dossier

Every target profile should maintain an evidence dossier before being marked supported.

Required fields:

- profile id
- device family
- claimed fidelity tiers
- vendor id, product id, version, bus type, and identity metadata
- known hardware revisions and firmware versions
- source links or local references for public evidence
- capture fixture ids for non-public evidence
- descriptor sources
- input report sources
- reverse output and feature report sources
- transport trace sources where applicable
- known gaps and assumptions
- validation status by tier

The dossier should distinguish verified behavior from inferred behavior.

## Discovery workflow

### Step 1: define the support question

Before collecting data, state the exact question.

Examples:

- what HID descriptor should this profile expose?
- which input report id carries buttons and axes?
- how does the host request rumble?
- does this device require a feature report during startup?
- does target software inspect USB enumeration rather than HID identity?

### Step 2: search public sources first

Preferred sources:

- official device or protocol documentation
- USB-IF, Bluetooth SIG, HID Usage Tables, and other standards documents
- Linux kernel drivers and HID quirks
- SDL gamecontroller mappings
- Steam Input behavior where publicly observable
- open-source drivers or emulation projects
- public HID descriptor databases
- community packet notes where licensing permits use

Output:

- source list
- confidence rating
- fields answered
- remaining unknowns

### Step 3: extract host-visible descriptors

For HID-capable targets, collect:

- HID report descriptor
- report ids
- input report lengths
- output report lengths
- feature report ids and lengths
- vendor/product/version metadata
- bus type metadata

Preferred Linux tools and sources:

- `/sys/bus/hid/devices`
- `/sys/kernel/debug/hid`
- `hid-recorder`
- `hid-decode`
- `lsusb -v`
- `udevadm info`
- `evtest`
- `libinput debug-events` where useful

Output:

- raw descriptor fixture
- decoded descriptor fixture
- identity metadata fixture
- parser validation result

### Step 4: capture representative input behavior

Capture one control group at a time.

Required baseline captures:

- neutral idle report
- each face button
- each d-pad direction and neutral
- both sticks at center, minimum, maximum, and diagonals
- analog triggers at minimum, midpoint, and maximum
- guide/start/back-style controls
- profile-specific controls such as touchpad, paddles, mute, motion sensors, or record/share keys

Output:

- raw input report fixtures
- decoded field map
- range and signedness notes
- unknown fields explicitly marked

### Step 5: capture reverse behavior

Reverse behavior must be captured before claiming `identity-aware` or `hardware-faithful` support.

Required captures where the device supports them:

- simple rumble
- independent rumble channels
- light or player LED changes
- adaptive trigger or haptic mode changes
- audio enable/mode commands
- feature report requests and responses
- unknown output reports generated by target software

Output:

- raw reverse report fixtures
- normalized expected `ControllerOutputCommand` fixtures
- unsupported or unknown report handling expectations

### Step 6: capture transport behavior only when required

Transport capture is needed only for `hardware-faithful` support or when target software rejects HID-level identity.

Required captures:

- connect/enumeration
- descriptor/control request sequence
- idle period
- representative input activity
- representative reverse command
- disconnect

Output:

- raw transport trace
- decoded packet model
- required startup state machine
- timing-sensitive observations
- differences tolerated at lower fidelity tiers

## Fixture conversion workflow

Every capture should be converted into executable fixtures.

Conversion steps:

1. store raw bytes exactly as captured
2. decode into a structured fixture
3. tag each field as verified, inferred, unknown, ignored, or reserved
4. link decoded fields to profile capabilities
5. create translator tests from the fixture
6. create reverse translator tests from output or feature reports
7. add planner/support-claim tests if the fixture exposes required fidelity behavior

Decoded fixtures should be small enough for review. Raw traces can be larger, but should be compressed or stored as external artifacts when needed.

## Validation matrix per device profile

Each profile should answer these questions before support is claimed:

| Area | Compatibility | Identity-aware | Hardware-faithful |
| --- | --- | --- | --- |
| Host-visible gameplay controls | required | required | required |
| Exact profile input contract | required | required | required |
| HID descriptor evidence | optional | required | required |
| HID input report evidence | optional | required | required |
| HID output report evidence | optional | required | required |
| HID feature report evidence | optional | required if used by target | required |
| USB/Bluetooth enumeration evidence | not required | not required | required |
| Transport control-flow evidence | not required | not required | required |
| Reverse packet evidence | optional | required at HID layer | required at transport layer |
| Target software recognition | recommended | required | required |

## Reverse-engineering minimization rules

- do not capture transport traffic if HID-level behavior satisfies the support question
- do not infer fields from noisy multi-control captures until single-control captures exist
- do not assign semantics to unknown fields just because they change
- prefer Linux driver behavior over private protocol guesses when the target is Linux host support
- use replay before full implementation to identify which fields are mandatory
- document every unknown field and retest it only when it blocks a support claim

## Public-source review checklist

Before real-device capture, verify whether public sources already answer:

- identity metadata
- descriptor contents
- report ids and lengths
- input field layout
- output report layout
- feature report layout
- initialization sequence
- reverse command semantics
- transport requirements
- known OS or firmware quirks

If a public source is sufficient, create fixtures from that source and record the source in the profile evidence dossier.

## Real-hardware capture checklist

When public sources are insufficient, collect:

- device model, hardware revision, firmware version if known
- host OS, kernel version, Bluetooth adapter or USB controller info where relevant
- capture tool versions
- raw descriptor or transport trace
- decoded fixture
- exact action performed for each capture
- expected normalized capability or command
- uncertainty notes

Captures should be reproducible by another developer with the same hardware.

## Support-claim gates

Before marking a profile as `compatibility`:

- profile input contract exists
- evdev capability and event fixtures pass
- host-visible virtual gamepad validation passes on Linux

Before marking a profile as `identity-aware`:

- HID descriptor evidence exists
- HID input report fixtures pass
- reverse output and feature report fixtures pass for the implemented subset
- target software or host tools recognize the virtual identity
- missing reverse behavior causes reject or degrade tests to pass

Before marking a profile as `hardware-faithful`:

- transport enumeration evidence exists
- control-flow and packet fixtures pass
- reverse packet fixtures pass
- real host accepts the virtual device at the transport layer
- differences from real hardware are documented and shown not to break the target scenario

## Final rule

A profile specification is not complete because the virtual controller can send inputs.

It is complete only when the device identity, forward reports, reverse reports, and required host-discovery behavior are backed by evidence and executable fixtures at the claimed fidelity tier.
