# Headless Test Strategy

This document defines how `virtualgamepad` should be validated in mostly remote, headless development environments while preserving a path to compare against real hardware on a dedicated validation machine.

The goal is to automate as much confidence as possible before any test requires a local human, physical controller, privileged kernel access, USB capture hardware, or Bluetooth capture setup.

## Validation principles

- treat real hardware as a fixture source, not the only test environment
- turn every observed descriptor, report, packet, and reverse command into replayable artifacts
- keep privileged and hardware-dependent tests optional, tagged, and skipped by default
- make every support claim reproducible from test output and captured evidence
- prefer deterministic fake backends before real kernel backends
- fail support claims when validation evidence is missing

## Environment tiers

### Tier A: ordinary remote CI

Runs without real devices, kernel modules, elevated privileges, or graphical display.

Required coverage:

- pure unit tests for profile input validation, capability metadata, config parsing, planner behavior, translators, and diagnostics
- contract tests for descriptor-to-translator consistency
- fake backend tests for session lifecycle, queue behavior, reverse-event delivery, and degradation
- replay tests from checked-in captures and fixtures
- documentation/spec consistency checks

This tier should be the default gate for every pull request.

### Tier B: privileged Linux runner

Runs on Linux with permission to use virtual-device APIs such as `uinput` and, later, `UHID`.

Required coverage:

- `uinput` device creation, capability declaration, evdev event emission, and teardown
- optional force-feedback event receipt when supported
- `UHID` device creation, HID descriptor provisioning, input report writes, output report receipt, feature report handling, and teardown
- host discovery through Linux inspection tools
- negative tests for permission failures and unavailable providers

This tier can run nightly or on demand because it depends on host configuration.

### Tier C: real-hardware capture runner

Runs on a controlled machine with actual target controllers and capture tools.

Required coverage:

- capture real descriptors and baseline identity metadata
- capture representative input reports or transport packets
- capture reverse reports generated by host software
- compare virtual-device behavior against captured real-device evidence
- refresh fixture artifacts when target firmware or host driver behavior changes

This tier is not required for every commit, but it is required before claiming or changing support for an `identity-aware` or `hardware-faithful` profile.

### Tier D: real-host compatibility bench

Runs against actual games, controller test tools, Steam/Input stacks, or other host software that motivated the profile.

Required coverage:

- host software discovers the virtual controller at the expected fidelity tier
- representative gameplay inputs work
- reverse features such as rumble, LEDs, trigger effects, audio, and mode commands are observed where claimed
- regression evidence is recorded as logs, traces, screenshots, or structured reports

This tier is manual-assisted or lab-automated, but its outputs should still become artifacts.

## Fixture artifact model

Every hardware or host-observed behavior should be stored as a small, typed fixture rather than only as prose.

Recommended fixture kinds:

- `identity`: vendor id, product id, version, bus type, device name, serial behavior where relevant
- `descriptor`: HID report descriptors, USB descriptors, Bluetooth SDP/GATT or equivalent records where relevant
- `input_report`: neutral report, button report, axis extremes, trigger extremes, d-pad states, motion samples, target-specific controls
- `reverse_report`: rumble, lighting, trigger effect, audio/mode, feature request/response, unknown report
- `transport_trace`: enumeration, control flow, idle packets, active input packets, reverse packets, disconnect
- `host_discovery`: what Linux, SDL, Steam, or other host software reports for the device

Fixture rules:

- include source device, firmware version when known, host OS/kernel/tool versions, capture date, and capture method
- prefer compact decoded JSON/YAML fixtures plus raw binary sidecars for byte-level comparison
- never overwrite old fixtures silently; version them when behavior changes
- mark proprietary or uncertain fields as unknown rather than inventing semantics

## Headless validation pipeline

The default automated pipeline should run in this order:

1. static checks for docs and schema consistency
2. unit tests for profile, capability, config, planner, translator, and telemetry logic
3. fake backend session tests for lifecycle and reverse-event delivery
4. descriptor/report fixture replay tests
5. degradation and support-claim tests
6. optional privileged Linux provider tests
7. optional hardware-capture comparison tests
8. optional real-host compatibility bench tests

Failure policy:

- Tier A failures block all merges
- Tier B failures block provider support claims
- Tier C failures block profile fidelity support claims
- Tier D failures block claims about compatibility with the specific host software under test

## Automation targets by fidelity tier

### `compatibility`

Automatable without hardware:

- evdev capability declarations match profile gameplay controls
- translated evdev event batches match expected axes, buttons, ranges, and sync order
- planner does not claim HID or transport identity

Requires privileged Linux:

- `/dev/input/event*` node appears
- Linux capability queries show expected keys and axes
- event stream matches submitted profile input
- teardown removes the device cleanly

### `identity-aware`

Automatable without hardware:

- HID descriptor parses successfully
- report ids, field widths, and translator output match descriptor fixtures
- reverse output and feature report fixtures parse into declared `ControllerOutputCommand`s
- missing reverse translator coverage fails support claims

Requires privileged Linux:

- UHID device appears with expected identity metadata
- host can read descriptor and receive input reports
- host-originated output and feature reports reach the backend and reverse translator

Requires real hardware before support claim:

- descriptor and report fixtures are derived from public specs, kernel drivers, or captures
- target software recognizes the virtual device at the claimed identity tier
- reverse commands observed from target software match normalized commands

### `hardware-faithful`

Automatable without hardware:

- transport state machines accept captured enumeration/control-flow fixtures
- packet encoders and decoders round-trip known traces
- planner rejects or degrades when transport support is unavailable

Requires transport-capable validation host:

- host enumerates the virtual device over the modeled bus
- control requests or protocol handshakes match required behavior
- reverse transport packets are observed and translated

Requires real hardware before support claim:

- startup, idle, active input, reverse command, and disconnect traces exist
- trace differences between real and virtual devices are either eliminated or documented as safe

## Test harness requirements

The implementation should eventually include a test harness that can:

- enumerate available providers and mark tests as skipped, failed, or unsupported with stable reason codes
- run the same profile fixture through fake, `uinput`, `UHID`, and transport backends where applicable
- replay raw reverse reports and transport packets into reverse translators
- record virtual-device discovery output from host tools
- export machine-readable support evidence for each claimed profile and fidelity tier

Recommended commands:

- `validate-fixtures`
- `replay-profile`
- `plan-session`
- `run-fake-session`
- `run-uinput-smoke`
- `run-uhid-smoke`
- `capture-real-device`
- `compare-real-device`
- `support-report`

## Support evidence report

Every profile/fidelity support claim should produce a report containing:

- profile id and profile family
- requested and selected fidelity tier
- backend provider used
- fixture set versions used
- tests passed, skipped, and failed
- enabled and unsupported capabilities
- reverse-output coverage summary
- real-hardware evidence status
- host-software evidence status where applicable

The report should be generated automatically so support claims can be reviewed without reading raw logs.

## Final rule

The remote/headless environment should catch design, contract, descriptor, report, planner, translation, queueing, and support-claim regressions.

The real-hardware environment should be reserved for producing authoritative fixtures and validating that the automated model still matches reality.
